# 小程序端与网页端开发计划

## 一、当前项目进展

当前系统已经具备第一版核心后端能力：

- 已有登录鉴权接口，登录后通过 `Authorization: Bearer <token>` 调用受保护接口。
- 已有客户管理接口，支持客户分页查询、详情、新增、编辑、删除。
- 已有板材管理接口，支持板材分页查询、详情、新增、编辑、删除。
- 已有算法求解接口 `/algorithm/answer`，可接收容器尺寸、旋转开关、间隙和矩形列表，返回排样结果。
- 已生成微信小程序基础前端资源，包含登录、客户、板材、算法输入和结果可视化页面。
- 已完成网页端第一版独立前端工程，包含登录、工作台、客户、板材、算法输入和结果可视化页面。
- 已提供本地开发启动/停止脚本和 Nginx 部署配置模板。
- 后端已补充参数校验、统一校验异常处理、日志治理、接口冒烟测试、登录鉴权测试和算法单元测试。

后续前端建设应避免一次性做得过重，建议按“小程序便捷录入、网页端管理分析”的定位分开推进。

## 二、小程序端开发计划

### 2.1 小程序端定位

小程序端优先服务现场、移动端和轻量场景，重点是降低录入成本，让用户可以快速完成客户、板材和待切尺寸录入，并立即调用算法查看排样结果。

小程序端未来的差异化能力是：**图片识别手写尺寸信息**。用户拍摄或上传手写尺寸单后，系统自动识别尺寸并生成待排矩形列表，减少逐条手动输入。

### 2.2 图片识别功能目标

第一阶段目标不是追求完全自动化，而是实现“识别 + 人工确认”的闭环：

1. 用户拍照或从相册选择图片。
2. 小程序上传图片到后端。
3. 后端调用 OCR/图像识别服务，识别手写尺寸文本。
4. 系统解析尺寸格式，例如 `800x600`、`800*600`、`800 600`、`长800 宽600`。
5. 小程序展示识别结果，并标注置信度或异常项。
6. 用户确认、修改或删除识别出的尺寸。
7. 确认后的尺寸自动填充到算法输入页的矩形列表。

### 2.3 推荐技术路线

#### 阶段一：后端 OCR 适配

新增后端能力：

- 图片上传接口，例如 `POST /recognition/dimensions`
- 请求类型：`multipart/form-data`
- 返回结构建议：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "rawText": "800x600 1200x500",
    "items": [
      {
        "l": 800,
        "w": 600,
        "confidence": 0.92,
        "sourceText": "800x600"
      }
    ],
    "warnings": []
  }
}
```

OCR 服务建议先抽象成接口：

- `OcrService`
- `DimensionParser`
- `RecognitionController`

这样后续可以在不改小程序的情况下替换识别供应商。

可选识别方案：

- 微信生态 OCR 能力，适合小程序闭环。
- 腾讯云 OCR、百度 OCR、阿里云 OCR，适合后端统一调用。
- 未来如需更强手写识别，可接入专门的多模态识别服务。

#### 阶段二：小程序图片识别页面

新增页面：

- `pages/recognition/index`

页面能力：

- 拍照/相册上传。
- 图片预览。
- 调用识别接口。
- 展示识别出的尺寸列表。
- 支持逐条编辑、删除、补充。
- 支持“一键导入算法输入页”。

建议不要把图片识别直接塞进算法页，而是做成独立页面，再跳转到算法页。这样算法页仍保持清晰，识别页可以独立迭代。

#### 阶段三：识别结果纠错与模板化

识别上线后，应优先提升纠错体验：

- 支持常见单位识别：`mm`、`cm`。
- 支持中英文符号：`x`、`X`、`*`、`×`。
- 支持批量修改单位。
- 支持异常尺寸提示，例如长度或宽度为 0、超出板材尺寸。
- 支持识别历史记录，方便用户复用。

### 2.4 小程序端功能优先级

#### P0：第一版可用

- 登录稳定。
- 客户列表、新增、编辑、详情可用。
- 板材列表、新增、编辑、删除可用。
- 算法输入和结果可视化可用。
- 请求失败、登录过期、参数错误有清晰提示。

#### P1：图片识别 MVP

- 新增图片识别入口。
- 支持拍照/选图。
- 后端 OCR 接口可用。
- 识别结果能转成尺寸列表。
- 用户确认后可导入算法输入页。

#### P2：识别体验优化

- 图片裁剪和旋转。
- 识别结果置信度展示。
- 低置信度项高亮。
- 支持多种尺寸书写格式。
- 支持识别失败后的手动补录。

#### P3：业务闭环增强

- 将识别结果关联客户。
- 将算法结果保存为方案。
- 支持历史方案查看。
- 支持方案导出或分享。

## 三、网页端开发计划

### 3.1 网页端定位

网页端不做图片识别功能。网页端应与小程序端形成不同定位：

- 小程序端：现场快速录入、拍照识别、即时排样。
- 网页端：后台管理、数据维护、批量操作、方案分析、权限管理。

网页端更适合承载复杂表格、批量导入、可视化分析和系统配置，不应重复小程序的拍照识别能力。

### 3.2 网页端核心功能

#### 第一阶段：后台基础管理

- 登录页。
- 客户管理页。
- 板材管理页。
- 算法输入和结果页。
- 统一请求封装。
- 登录态和 token 管理。

前端技术可选：

- Vue 3 + Vite + Element Plus
- React + Vite + Ant Design

考虑到管理系统表格和表单较多，建议优先选择 Vue 3 + Element Plus，开发成本较低。

#### 第二阶段：业务增强

- 客户筛选、搜索、分页。
- 板材按品牌、材质、颜色、尺寸筛选。
- 算法结果可视化增强。
- 排样方案保存。
- 历史方案列表和详情。

#### 第三阶段：管理与分析

- 用户管理。
- 角色权限管理。
- 板材使用频次统计。
- 常用尺寸统计。
- 算法利用率统计。
- 导出 Excel/PDF。

### 3.3 网页端不做图片识别的原因

- 图片识别更符合移动端拍照场景。
- 网页端用户通常处于办公场景，更适合批量录入、表格导入和方案管理。
- 避免两个端重复维护同一套复杂识别交互。
- 有利于形成清晰产品定位：小程序解决录入便捷性，网页端解决管理效率。

## 四、开发节奏建议

### 第一阶段：稳定当前基础能力

目标：让小程序现有功能能真实跑通。

重点任务：

- 确认后端数据库和测试账号。
- 联调小程序登录、客户、板材、算法接口。
- 修正小程序字段、错误提示和页面跳转问题。
- 补充小程序端基础异常处理。
- 确认真机调试访问地址和后端跨端访问方式。

建议周期：1 周。

### 第二阶段：小程序图片识别 MVP

目标：完成从图片到尺寸列表的最小闭环。

重点任务：

- 后端新增图片上传和 OCR 识别接口。
- 后端实现尺寸文本解析。
- 小程序新增图片识别页面。
- 小程序支持识别结果编辑和导入算法页。
- 增加识别接口测试和解析单元测试。

建议周期：2 周。

### 第三阶段：网页端第一版

状态：已完成第一版。

目标：提供稳定的后台管理入口。

重点任务：

- 搭建网页端工程（已完成）。
- 实现登录、客户、板材、算法页面（已完成）。
- 统一接口封装和 token 管理（已完成）。
- 实现表格分页和基础筛选（分页与基础表单已完成，复杂筛选待增强）。
- 与小程序共用后端接口。

建议周期：2 周。

### 第四阶段：方案保存与业务闭环

目标：从“能算”升级为“能管理方案”。

重点任务：

- 后端新增方案表和方案接口。
- 小程序支持保存识别结果和算法方案。
- 网页端支持方案列表、详情、筛选、导出。
- 统计板材使用频次和空间利用率。

建议周期：2 到 3 周。

## 六、风险与注意事项

- 手写 OCR 准确率不可控，必须保留人工确认和编辑。
- 真机调试时不能使用 `localhost`，需要配置局域网 IP 或正式域名。
- 图片上传会引入文件大小、格式、安全校验和存储清理问题。
- 算法接口耗时可能随矩形数量增长，需要后续考虑超时控制或异步任务。
- 网页端和小程序端应共用后端接口，但不要强行共用所有交互形态。

---

## 七、结论

建议项目后续采用“双端差异化”路线：

- 小程序端重点做轻量录入、图片识别、现场快速排样。
- 网页端重点做后台管理、数据维护、方案分析和导出。

短期优先保证当前小程序基础功能稳定，再推进图片识别 MVP；网页端第一版已经完成，后续应围绕排样方案保存、历史排单、余料库和统计分析逐步增强，避免两个端同时扩张导致维护压力过大。
