# 文档总览

整理日期：2026-04-30

本文档用于统一说明当前项目文档结构，提取历史文档中的有效信息，并标记已经清理的低价值内容。后续查阅项目资料时，优先从本文档和根目录 `README.md` 开始。

## 1. 当前项目结论

项目定位已经从早期“柜门加工系统”原型，收敛为“板材切割系统”。两者业务内核一致：围绕客户、板材、余料、排单、尺寸录入、算法排样和结果复用，降低人工录入和排样成本，提高板材利用率。

当前工程已经具备以下可用基础：

- Spring Boot 后端服务。
- JWT 登录鉴权。
- 客户、板材基础管理。
- 算法求解接口。
- Vue 3 网页端第一版。
- 微信小程序基础页面。
- 本地开发启动/停止脚本。
- 数据库补齐 SQL 草案。
- 算法模块设计文档。

后续建设应围绕“业务闭环”推进：新建排单、排样方案保存、历史排单复用、余料库、统计分析和小程序图片识别。

## 3. 已提取的有效信息

### 3.1 产品与业务

有效信息来自历史产品需求、柜门加工构想和项目说明：

- 目标用户包括生产人员、排单员、加工管理人员和管理员。
- 生产人员侧重点是客户、板材、余料、排单、排样和历史复用。
- 管理员侧重点是用户管理、账号启用/禁用和系统配置。
- 小程序适合现场快速录入、拍照识别和即时排样。
- 网页端适合后台管理、复杂表格、批量导入、方案分析和导出。
- 图片识别必须采用“识别 + 人工确认”模式，不能直接全自动入库。

### 3.2 当前已实现能力

有效信息来自现有代码和项目说明：

- 后端：登录、注册、鉴权拦截、客户管理、板材管理、算法求解、参数校验、统一异常、日志、测试。
- 网页端：登录、工作台、客户管理、板材管理、算法输入、Canvas 可视化结果。
- 小程序端：登录、客户、板材、算法输入、结果表格和 canvas 展示。
- 算法：禁忌搜索优化矩形顺序，天际线算法计算放置坐标，多容器迭代输出方案。
- 开发工具：`scripts/start-dev.ps1` 和 `scripts/stop-dev.ps1`。

### 3.4 数据模型

当前数据库已具备基础表：

- `t_user`
- `t_board`
- `t_customer`
- `t_offcut`
- `t_order`
- `t_order_item`

当前已整理出的目标补齐方向：

- 补齐板材、客户、余料、订单等表的用户维度外键和索引。
- 为排单保存业务快照字段，如 `raw_material_json`、`remnant_json`、`config_json`。
- 新增 `t_layout_result`，保存排版结果 JSON、利用率、容器数量和导出文件路径。
- 相关 SQL 已放在 `src/main/resources/db/migration/20260430_align_bishe_schema.sql`。

### 3.5 接口约定

当前后端接口以 REST 风格为主：

- `POST /auth/login`
- `POST /auth/register`
- `GET/POST/PUT/DELETE /customers`
- `GET/POST/PUT/DELETE /boards`
- `POST /algorithm/answer`

建议后续新增：

- `GET /auth/me`
- `GET /remnants`
- `POST /orders`
- `POST /orders/{id}/layout`
- `GET /layout-results/{id}`
- `POST /recognition/dimensions`
- `GET /admin/users`

### 3.6 算法模块

有效信息已整理到 `src/main/resources/相关文档/算法模块整体概况.md`：

- 禁忌搜索：优化矩形排列顺序。
- 天际线算法：根据排列顺序计算放置坐标。
- 禁忌映射树：记录已访问排列，减少重复搜索。
- 多容器迭代：一个容器装不完时继续求解下一容器。
- 单件兜底评估：主搜索无进展时仍尝试放入至少一个可行矩形。

仍需处理的算法风险：

- `squareList` 缺少数量上限。
- 容器、矩形和间隙缺少业务最大值约束。
- 随机搜索结果不可完全复现。
- 当前评分只考虑面积利用率。
