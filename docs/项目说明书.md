# 板材切割系统项目说明书

版本：V1.0  
日期：2026-04-28

## 一、项目背景

板材切割业务通常涉及客户信息维护、板材规格管理、待切尺寸录入、排样计算和结果查看等流程。传统方式中，尺寸信息往往通过手写单、口头记录或表格传递，人工录入工作量较大，容易出现漏录、错录和重复录入等问题。同时，板材利用率直接影响材料成本，排样方案需要尽可能提高空间利用率并支持快速查看结果。

当前项目已经完成后端基础能力、小程序基础页面、核心算法接口和网页端第一版建设。系统后端基于 Spring Boot 实现，已具备登录鉴权、客户管理、板材管理、算法求解、参数校验、日志治理和测试覆盖等基础能力。小程序端已生成登录、客户、板材、算法输入与结果可视化页面资源。网页端已采用 Vue 3 + Vite 独立工程实现登录、客户管理、板材管理、算法输入与结果可视化，并提供本地开发启动脚本和 Nginx 部署模板。后续将围绕“小程序轻量录入与图片识别、网页端后台管理与分析、排样方案保存和历史复用”的方向继续扩展。

## 二、项目目标

### 2.1 总体目标

建设一套面向板材切割业务的管理与排样系统，实现客户、板材、尺寸录入、算法求解和结果展示的一体化流程，降低人工录入成本，提高板材利用率和业务处理效率。

### 2.2 阶段目标

第一阶段目标：

- 完成后端核心接口稳定化。
- 完成小程序基础功能联调。

- 通过可视化方式展示算法计算结果。

第二阶段目标：

- 在小程序端引入图片识别能力。
- 支持识别用户手写尺寸信息。
- 将识别结果转为可编辑的尺寸列表。
- 支持用户确认后导入算法输入页。

第三阶段目标：

- 网页端第一版后台管理系统已完成。
- 网页端已聚焦数据维护、算法输入和结果展示，不引入图片识别功能。
- 后续继续扩展批量操作、方案管理、历史排单和统计分析。

第四阶段目标：

- 建立排样方案保存和历史查询能力。
- 支持方案导出、利用率统计和板材使用分析。
- 支撑后续角色权限、生产流程和多端协同扩展。

## 四、系统定位

### 4.1 小程序端定位

小程序端面向移动端和现场场景，强调轻量、快速和便捷。其核心价值是降低尺寸录入成本，使用户能够快速完成客户选择、板材选择、尺寸录入和排样查看。

小程序端后续重点能力是图片识别手写尺寸：

- 用户拍照或从相册选择手写尺寸单。
- 系统识别尺寸文本。
- 用户确认识别结果。
- 一键导入算法输入页。

### 4.2 网页端定位

网页端面向后台管理和办公场景，强调信息维护、批量处理、方案分析和数据统计。

网页端不包含图片识别功能，原因如下：

- 图片识别更符合手机拍照场景。
- 网页端更适合复杂表格、批量导入、筛选和导出。
- 双端职责分离可以降低维护成本。
- 小程序解决录入便捷性，网页端解决管理效率。

## 五、技术方案

### 5.1 后端技术方案

后端采用 Java + Spring Boot 架构，当前主要技术组件如下：

- Spring Boot 3.x：应用框架。
- Spring Web：REST 接口。
- MyBatis-Plus：数据访问和分页。
- MySQL：业务数据存储。
- JWT：登录鉴权。
- Jakarta Bean Validation：请求参数校验。
- Slf4j + Logback：日志输出。
- JUnit 5 + MockMvc：测试覆盖。

后端主要模块：

- `AuthController`：登录、注册、登出。
- `CustomerController`：客户管理。
- `BoardsController`：板材管理。
- `TestController`：算法求解接口。
- `TabuSearch`：禁忌搜索与天际线排样算法。
- `ReadDataUtil`：算法输入转换和多容器求解流程。
- `TokenInterceptor`：token 校验和用户上下文设置。

接口规范：

- 查询使用 `GET /resources` 或 `GET /resources/{id}`。
- 新增使用 `POST /resources`。
- 更新使用 `PUT /resources/{id}`。
- 删除使用 `DELETE /resources/{id}`。
- 登录接口使用 `POST /auth/login`。
- 算法接口使用 `POST /algorithm/answer`。

### 5.2 算法技术方案

当前算法使用“禁忌搜索 + 天际线放置”的启发式方案：

- 天际线算法负责根据给定矩形顺序进行二维排样。
- 禁忌搜索通过随机交换矩形顺序寻找更高利用率方案。
- 支持是否允许 90 度旋转。
- 支持矩形间隙距离。
- 支持多容器迭代求解。

当前算法适合作为第一版可用能力，但仍需后续优化：

- 增加大规模输入的耗时控制。
- 支持无法放入矩形的明确提示。
- 返回放置结果与原始尺寸 ID 的关联。
- 支持方案保存和复现。
- 后续可引入更多排样策略进行对比。

### 5.3 小程序端技术方案

小程序端采用微信小程序原生开发方式，当前结构如下：

- `app.js`：全局 token 和用户信息初始化。
- `app.json`：页面和 tabBar 配置。
- `app.wxss`：全局样式。
- `utils/request.js`：统一请求封装。
- `utils/config.js`：后端地址配置。
- `services/api.js`：业务接口封装。
- `pages/login`：登录页。
- `pages/customers`：客户模块。
- `pages/boards`：板材模块。
- `pages/algorithm`：算法输入和结果可视化模块。

小程序请求策略：

- 登录成功后保存 token。
- 后续业务请求统一携带 `Authorization: Bearer <token>`。
- 401 自动清理登录态并跳转登录页。
- 接口错误统一提示。

### 5.4 图片识别技术方案

图片识别功能建议采用“后端统一适配 OCR 服务，小程序只负责上传与确认”的方式。

建议新增后端模块：

- `RecognitionController`：图片识别接口。
- `OcrService`：OCR 服务抽象。
- `DimensionParser`：尺寸文本解析器。
- `RecognitionResultDTO`：识别结果返回对象。

建议接口：

```http
POST /recognition/dimensions
Content-Type: multipart/form-data
```

返回示例：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "rawText": "800x600 1200x500",
    "items": [
      {
        "l": 800,
        "w": 600,
        "confidence": 0.92,
        "sourceText": "800x600"
      }
    ],
    "warnings": []
  }
}
```

识别流程：

1. 小程序拍照或选择图片。
2. 小程序上传图片到后端。
3. 后端调用 OCR 服务。
4. 后端解析尺寸文本。
5. 小程序展示识别结果。
6. 用户编辑和确认。
7. 导入算法输入页。

可选 OCR 服务：

- 微信生态 OCR。
- 腾讯云 OCR。
- 百度 OCR。
- 阿里云 OCR。
- 后续可接入多模态识别服务。

### 5.5 网页端技术方案

网页端建议采用 Vue 3 + Vite + Element Plus。

当前网页端第一版已经采用 Vue 3 + Vite + Pinia + Vue Router + Axios 实现。考虑第一版功能边界较轻，暂未引入 Element Plus，而是使用自定义基础样式和组件。后续如果表格、表单、弹窗和筛选复杂度继续上升，可再引入 Element Plus 或其他组件库。

推荐原因：

- 适合后台管理系统。
- 表格、表单、分页、弹窗等组件成熟。
- 开发效率较高。
- 与当前 REST 接口对接简单。

网页端模块建议：

- 登录模块（已完成）。
- 客户管理模块（已完成）。
- 板材管理模块（已完成）。
- 算法输入与结果展示模块（已完成基础版）。
- 排样方案管理模块。
- 数据统计模块。
- 系统设置与用户管理模块。

网页端不实现图片识别，只保留批量录入、表格导入和方案分析能力。

## 八、风险说明

### 8.1 识别准确率风险

手写尺寸识别受字迹、光照、拍摄角度、纸张背景和单位格式影响，准确率不可完全保证。

应对措施：

- 采用“识别 + 人工确认”模式。
- 展示原始识别文本。
- 支持逐条编辑和删除。
- 对低置信度项进行标记。
- 优先支持常见格式，再逐步扩展。

### 8.2 算法性能风险

随着矩形数量增加，启发式搜索耗时可能增长，接口可能出现响应变慢。

应对措施：

- 设置输入数量上限。
- 增加算法耗时日志。
- 后续支持异步任务。
- 后续支持算法参数配置。
- 对典型业务场景进行性能测试。

### 8.3 数据准确性风险

客户、板材和尺寸数据如果录入错误，会直接影响算法结果和业务决策。

应对措施：

- 保留后端参数校验。
- 前端提供必要的输入提示。
- 识别结果必须人工确认。
- 后续增加方案保存前的校验流程。

### 8.4 真机调试与网络风险

小程序真机调试不能使用 `localhost`，需要使用局域网 IP 或正式域名。正式上线还需要 HTTPS 域名和微信小程序后台域名配置。

应对措施：

- 开发阶段使用局域网 IP。
- 配置小程序开发工具本地调试设置。
- 上线前准备 HTTPS 域名。
- 后端配置跨端访问和部署环境。

### 8.5 双端维护风险

小程序端和网页端如果功能重叠过多，会增加开发和维护成本。

应对措施：

- 明确双端定位。
- 小程序聚焦移动录入和图片识别。
- 网页端聚焦管理、分析和导出。
- 共用后端接口，避免重复建设业务逻辑。

### 8.6 安全风险

当前系统涉及登录鉴权、图片上传、业务数据管理等能力，后续需要加强安全控制。

应对措施：

- 登录密码存储后续升级为 BCrypt。
- 图片上传增加大小、类型和内容校验。
- 增加角色权限控制。
- 生产环境中 JWT 密钥和数据库密码使用环境变量。
- 关闭生产环境 SQL 明文日志。

## 十、总结

本项目当前已经具备板材切割业务的基础后端能力和小程序端原型资源。后续建议继续坚持“双端差异化”路线：

- 小程序端承担现场快速录入、图片识别和即时排样。
- 网页端承担后台管理、批量维护、方案分析和统计导出。

短期应优先完成小程序与后端真实联调，确保基础业务闭环稳定；随后推进图片识别 MVP；网页端则在核心业务流程稳定后启动建设。该节奏可以在控制复杂度的同时，逐步形成可用、可扩展、可维护的板材切割管理系统。
