# 算法模块整体概况

本文档用于说明当前算法模块的整体设计、使用的算法、实现方式、功能作用，以及相关数据结构、参数配置和性能优化方法。

## 1. 模块定位

当前算法模块主要服务于二维矩形装箱问题。接口接收一个容器尺寸和一组待切割/待装入矩形，根据配置判断是否允许旋转、是否需要保留间隙，然后计算每个矩形在容器中的放置位置。

当一个容器无法装下全部矩形时，模块会采用多容器迭代策略：先计算当前容器可装入的矩形，移除已装入矩形，再继续对剩余矩形求解，直到所有矩形处理完成或发现存在无法装入的矩形。

核心入口和实现文件如下：

| 文件 | 作用 |
| --- | --- |
| `controller/TestController.java` | 提供 `/algorithm/answer` 接口 |
| `util/ReadDataUtil.java` | 请求数据转换、多容器迭代、结果整理 |
| `model/TabuSearch.java` | 禁忌搜索和天际线评价算法核心 |
| `model/TabuMapTree.java` | 禁忌表树结构 |
| `entitys/algorithm/*` | 算法输入、输出和中间结果实体 |

## 2. 当前使用的算法

当前模块主要包含以下算法或策略：

| 算法/策略 | 所在位置 | 主要作用 |
| --- | --- | --- |
| 禁忌搜索 Tabu Search | `TabuSearch.search()` | 优化矩形排列顺序，寻找更高空间利用率 |
| 天际线 Skyline 放置算法 | `TabuSearch.evaluate()` | 在给定排列顺序下计算具体放置坐标 |
| 禁忌映射树 | `TabuMapTree` | 记录已访问过的矩形排列，避免重复搜索 |
| 多容器迭代拆分 | `ReadDataUtil.getSolution()` | 将全部矩形拆分到一个或多个容器中 |
| 单件兜底评估 | `ReadDataUtil.findBestSingleSquareSolution()` | 当主搜索无进展时，尝试找出至少一个可放入矩形 |

## 3. 算法实现方式

### 3.1 禁忌搜索

禁忌搜索是当前模块的顺序优化算法，使用自定义 Java 逻辑实现，没有引入第三方优化库。

它不直接优化矩形坐标，而是优化矩形的排列顺序。每个候选排列会交给天际线算法进行放置和评分，评分指标为当前容器的空间利用率。

主要流程：

1. 从输入矩形列表复制出初始排列。
2. 使用 `Collections.shuffle` 随机打乱，生成初始解。
3. 在每轮迭代中生成多个邻域解。
4. 邻域解通过随机交换矩形位置生成。
5. 对未进入禁忌表的排列调用 `evaluate()` 计算利用率。
6. 记录当前领域最优解和全局最优解。
7. 达到最大迭代次数后返回全局最优解。

关键配置：

| 参数 | 当前值 | 含义 |
| --- | --- | --- |
| `MAX_GEN` | `300` | 最大迭代次数 |
| `N` | `500` | 每轮邻域搜索次数 |
| 随机交换次数 | `6` | 生成新排列时执行的交换次数 |
| 评分指标 | `usedArea / totalArea` | 已放置矩形面积占容器面积比例 |

### 3.2 天际线放置算法

天际线算法是当前模块的核心评价函数，用于在给定矩形排列顺序下计算实际放置方案。

它维护一组天际线线段 `SkylineSegment`，每个线段表示当前容器中某段水平范围的可用高度。

`SkylineSegment` 包含：

| 字段 | 含义 |
| --- | --- |
| `x` | 线段起始 X 坐标 |
| `y` | 当前线段高度 |
| `width` | 当前线段宽度 |

放置流程：

1. 初始化天际线，起点和宽度会考虑 `gapDistance`。
2. 按候选矩形顺序逐个尝试放置。
3. 如果允许旋转，则同时尝试原始方向和旋转方向。
4. 扫描所有天际线线段，寻找满足容器边界和间隙约束的位置。
5. 优先选择高度更低的位置；高度相同时选择更靠左的位置。
6. 成功放置后生成 `PlaceSquare`。
7. 更新天际线：切分被占用线段、添加新顶部线段、合并同高度连续线段。
8. 全部矩形处理完成后计算利用率。

输出结果包括：

| 输出 | 含义 |
| --- | --- |
| `placeSquareList` | 已成功放置的矩形坐标和尺寸 |
| `rate` | 当前容器空间利用率 |
| `squareList` | 当前求解使用的候选矩形顺序 |

### 3.3 禁忌映射树

`TabuMapTree` 是禁忌搜索使用的数据结构，用于记录已经访问过的矩形排列。

它类似 Trie 树，使用矩形 ID 作为路径节点。例如一个排列：

```text
squareA -> squareB -> squareC
```

会被存储为一条树路径。后续如果再次生成相同排列，就可以通过递归查找判断该排列是否已访问，从而跳过重复评价。

主要方法：

| 方法 | 作用 |
| --- | --- |
| `add(List<Square>, int)` | 将一个排列从指定索引开始加入禁忌树 |
| `contains(List<Square>, int)` | 判断一个排列是否已存在于禁忌树 |
| `show()` | 递归遍历树结构，用于调试 |

### 3.4 多容器迭代拆分

多容器逻辑由 `ReadDataUtil.getSolution()` 实现。

单次 `TabuSearch` 只负责一个容器的求解。当仍有未装入矩形时，`ReadDataUtil` 会继续构造新的 `Instance`，再次调用禁忌搜索。

主要流程：

1. 将接口输入转换为内部 `Instance`。
2. 为每个矩形生成内部 UUID，作为后续精确匹配依据。
3. 使用当前剩余矩形构造一个容器实例。
4. 调用 `TabuSearch.search()` 求解当前容器。
5. 根据求解结果中的已放置矩形，从剩余列表中移除对应矩形。
6. 如果当前轮没有移除任何矩形，则进入单件兜底评估。
7. 如果兜底仍无法放入任何矩形，则抛出异常，避免死循环。
8. 重复以上步骤，直到剩余矩形为空。

### 3.5 单件兜底评估

单件兜底评估用于处理主搜索没有产生有效装箱结果的情况。

实际业务中，无法装入容器的原因不一定只有尺寸超限，也可能是评价结果过低、搜索过程没有找到合适排列、或者某一轮候选解没有产生有效进展。

因此当前逻辑在发现 `removedCount == 0` 时，会逐个尝试剩余矩形：

1. 将单个矩形构造成独立 `Instance`。
2. 调用天际线评价函数 `evaluate()`。
3. 如果可以放入，则选择利用率最高的单件方案作为本轮结果。
4. 如果没有任何矩形可放入，则抛出异常。

这个策略的目标是保证：只要还存在至少一个可放入容器的矩形，算法就能继续推进，不会因为主搜索评分过低而卡死。

## 4. 数据结构说明

| 数据结构 | 所在类 | 作用 |
| --- | --- | --- |
| `Instance` | `entitys/algorithm/Instance.java` | 描述一次装箱问题，包括容器尺寸、旋转配置、间隙和矩形列表 |
| `Square` | `entitys/algorithm/Square.java` | 待装入矩形，包含 ID、长、宽 |
| `Solution` | `entitys/algorithm/Solution.java` | 算法求解结果，包含利用率、原始矩形列表和已放置矩形列表 |
| `PlaceSquare` | `entitys/algorithm/PlaceSquare.java` | 已放置矩形，包含坐标和尺寸 |
| `SkylineSegment` | `TabuSearch` 内部类 | 天际线线段，用于表示当前容器的可用轮廓 |
| `TabuMapTree` | `model/TabuMapTree.java` | 禁忌表树结构，记录已访问排列 |
| `HashMap<String, TabuMapTree>` | `TabuSearch.tabuTreeMap` | 按首个矩形 ID 分组存储禁忌路径 |
| `HashSet<String>` | `ReadDataUtil.removePackedSquares()` | 已匹配矩形 ID 集合，防止同尺寸矩形误删 |

## 5. 输入参数和算法配置

接口输入主要来自 `InstanceDTO`。

| 参数 | 类型 | 作用 |
| --- | --- | --- |
| `L` | `double` | 容器长度 |
| `W` | `double` | 容器宽度 |
| `rotateEnable` | `boolean` | 是否允许矩形旋转 90 度 |
| `gapDistance` | `double` | 矩形和边界之间保留的间隙 |
| `squareList` | `List<Square>` | 待装入矩形列表 |

矩形参数：

| 参数 | 类型 | 作用 |
| --- | --- | --- |
| `id` | `String` | 外部传入的矩形标识，当前内部会重新生成 UUID |
| `l` | `double` | 矩形长度 |
| `w` | `double` | 矩形宽度 |

算法内部配置：

| 配置 | 当前值 | 影响 |
| --- | --- | --- |
| `MAX_GEN` | `300` | 值越大，搜索更充分，但耗时更高 |
| `N` | `500` | 每轮邻域数量，值越大，单轮搜索成本越高 |
| 随机种子 | `System.currentTimeMillis()` | 每次求解结果可能存在随机差异 |
| 评分函数 | 空间利用率 | 只按面积利用率评价，不直接考虑切割路径、材料纹理等因素 |

## 6. 性能优化和稳定性处理

当前已有处理：

1. 使用禁忌映射树记录已访问排列，减少重复搜索。
2. 使用天际线结构增量维护可放置轮廓，不需要每次扫描完整二维网格。
3. 通过 `MAX_GEN` 和 `N` 限制搜索规模，避免搜索无限扩张。
4. 多容器求解时，每轮移除已装入矩形，逐步缩小后续问题规模。
5. 已装入矩形按内部 UUID 精确移除，避免相同尺寸矩形误删。
6. 当一轮没有装入任何矩形时，执行单件兜底评估，避免主搜索评分过低导致流程停滞。
7. 兜底仍无解时抛出异常，避免接口无限循环。

仍需关注的点：

1. `squareList` 当前缺少数量上限，超大输入可能导致 CPU 和内存压力。
2. 容器尺寸、矩形尺寸和间隙距离缺少业务最大值约束。
3. 禁忌搜索使用随机扰动，相同输入在不同运行中可能出现不同结果。
4. 当前评分函数只考虑面积利用率，尚未考虑切割工艺、余料形状、刀路成本等业务目标。
5. `TabuSearch.evaluate()` 中相邻天际线段合并利用仍有优化空间。

## 7. 总体流程

```text
前端/调用方
  |
  | POST /algorithm/answer
  v
TestController
  |
  | InstanceDTO -> JSON
  v
ReadDataUtil
  |
  | JSON -> Instance
  | 生成内部矩形 UUID
  | 构造当前容器实例
  v
TabuSearch.search()
  |
  | 生成候选排列
  | 禁忌表去重
  | 调用 evaluate() 评分
  v
TabuSearch.evaluate()
  |
  | 天际线放置
  | 计算利用率
  v
Solution
  |
  | 移除已装入矩形
  | 如仍有剩余，继续下一个容器
  v
List<SolutionResponseDTO>
```

## 8. 适用场景

当前算法适合用于：

1. 二维矩形板材排布。
2. 简化版装箱/套料问题。
3. 需要支持多容器输出的矩形装载任务。
4. 可以接受启发式结果，而不要求严格数学最优解的业务场景。

不太适合直接用于：

1. 必须保证全局最优的精确求解场景。
2. 需要复杂切割路径、刀缝、纹理方向、封边规则的生产级套料。
3. 大规模高并发求解，除非继续增加输入规模限制、任务队列、超时控制和缓存策略。
